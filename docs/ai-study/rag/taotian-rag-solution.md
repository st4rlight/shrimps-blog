# 淘天RAG方案

> RAG 全链路技术详解

## 文档处理

### 文档加载

将各种各样非结构化的数据转化为系统可以处理的数据。

- **多格式适配兼容**：文件来源可能多种多样，类型也是
- **内容提取**：从原始文件中提取出信息，可能需要 ASR、OCR、剔除样式等操作
- **元数据提取**：提取文档的同时，记录额外的文档属性信息，如文件名、页码、标题、作者、创建时间、章节层级等
- **初步清洗**：去除文档中的冗余信息，比如乱码、多余的空格、特殊打印字符等

### 文档切分

| 分块方法 | 方法描述 | 优点 | 缺点 | 适用场景 |
| --- | --- | --- | --- | --- |
| 按token数分块 | 预先设定一个固定的字符数或 Token 数（如 512 个 Token），将文档机械地拆分为等长的块，通常会设置一定的重叠度 (Overlap) 以减少连接处的信息丢失。 | 实现极其简单，计算开销最小，不依赖任何模型。 | 完全忽略语义结构，极易在句子或段落中间断开，导致语义碎片化。 | 对性能要求极高、处理超大规模无结构文本，且对检索精度要求不高的初步实验。 |
| 递归字符切块 | 通过一组层级分隔符（如 `['\n\n', '\n', ' ', '']`）递归地拆分文本。它首先尝试按段落拆分，如果段落太大，再按行拆分，以此类推，直到切块大小符合要求。 | 能尽可能保持段落和句子的完整性，比固定切块更贴合人类阅读习惯。 | 依然是一种规则驱动的硬切，对于极其复杂的文档结构，仍可能丢失深层语境。 | 通用的 PDF、Word 文档、普通文章，是大多数 RAG 项目的默认首选。 |
| 基于结构的切块 | 针对特定格式的文档，利用其自身的语法标记进行切块。解析 Markdown、HTML、JSON 的语法树。例如，Markdown 按标题层级 (H1, H2, H3) 切块。 | 完美保留文档的逻辑层级，检索到的内容通常具有极高的自解释性。 | 高度依赖输入文档的格式规范性。如果 Markdown 格式混乱，效果会大打折扣。 | 技术文档、API 文档、代码库、结构化的 Web 页面 |
| 按语义分块 | 基于文本背后的"意思"而非物理符号来决定切分点。先将文档拆分为句子，利用 Embedding 模型将句子向量化。计算相邻句子间的余弦相似度，当相似度低于某个阈值（说明话题转折了）时，进行切块。 | 保证了每个块内的语义高度统一，极大提升了检索的相关性。 | 由于需要对所有句子进行向量化计算，计算成本高，处理速度比规则切块慢得多。 | 长篇报告、叙事性文学、话题转换频繁的深度内容。 |

## 检索查询

### Query 改写

- **核心目标**：对用户原始查询进行优化，纠正原始问题中的偏差、模糊和不完整的信息
- **解决的问题**：
  - **上下文关联**：多轮对话场景下改写用户的问题描述
  - **纠错与去除噪音**：修正错别字，剔除无意义的口语助词
  - **术语对齐**：将口语化的表达转为专业术语（例如将"肚子疼怎么办"改写为"腹痛的治疗建议"）
  - **结构转换**：将复杂句子拆解简化，提取出核心语义特征

### 多查询生成

- 将用户问题转成 3-5 个语义相近但是表达不同的问题，分别检索，提升召回率
- 一个示例：**如何提高睡眠质量**
  - 睡前哪些习惯有助于入睡？（侧重生活习惯）
  - 什么样的卧室环境适合深度睡眠？（侧重环境问题）
  - 改善失眠的科学方法有哪些？（侧重医学建议）
- **效果**：从多个角度出发改写问题，防止用户提问过于片面而漏掉关键资料

### HyDE：用假设文档增强检索

> 有时候系统很难凭借一个短小的问题就精准捕捉到长篇大论的答案

- **Hypothetical Document Embeddings**
- 先让 AI 编一个答案，再用这个答案去搜真内容
- 本质是将"问题-文档"匹配转换成"文档-文档"匹配

### 反向 HyDE（Doc2Query）

- 针对 chunk，生成这块 chunk 可能的 question，然后针对这些 question 进行索引构建，关联到具体的 chunk 内容
- 反向 HyDE 相比 HyDE 的优势是**可以离线处理，不影响实时调用的 RT**
- 一个示例：
  - 原始文本：*"根据公司差旅管理办法第四条，员工在非一线城市出差时，每日住宿补贴上限为 350 元人民币。申请人需在差旅结束后 15 个工作日内，通过 ERP 系统提交正规增值税专用发票进行报销，逾期将不予受理。"*
  - 让大语言模型针对这段话生成 3-5 个"虚构问题"：
    1. 出差去二线城市住酒店一天能报销多少钱？
    2. 出差回来后最晚什么时候报销？
    3. 报销差旅费需要什么类型的发票？
  - 在知识库中，将这段"原始文本"和这 3 个"虚构问题"关联在一起并建立索引

### 提取标签

> 标签化之后能够显著降低一些关联性较大的噪音

- 是一种将"非结构化数据"向"半结构化数据"转化的过程。它通过在语义向量检索的基础上引入标签过滤来提升检索精度。
- 例子：
  - 用户搜"iPhone 15 Pro Max"。向量可能会把它和"iPhone 14"放在一起（因为语义很像），但标签匹配能通过 iphone15 这种硬标签精准锚定目标。
  - **效果**：过滤掉大量"语义接近但事实不符"的噪音，解决了"看着像但不是同一个东西"的问题，增加了检索的确定性。

### 重排序（ReRank）

> 检索所返回出来的 docs，实际上可能部分和用户 query 关联度并不大
>
> 在文档的索引阶段，文档的向量是预先计算并存好的，与 Query 无关，这导致它只能表达文档的平均含义

- 通过 ReRank 步骤，对 docs 做重排序，最终选取最相关的 top k 个 doc
- **原理**：
  - 利用交叉编码器（Cross-Encoder），把问题（Query）和"候选文档"拼在一起，作为一个长句子塞进模型（比如：gte-rerank）
  - 模型可以同时看到问题和文档的每一个字。通过自注意力机制（Self-Attention），在 Transformer 的每一层中，Query 中的每一个 token 都可以直接通过 Attention 机制观察到文档中的每一个词
  - 这种交叉注意力（Cross-Attention）能够捕捉到极细微的匹配关系，判定问题与文档的匹配程度
- ReRank 也是需要模型参与，不过是专门的 ReRank 模型，将文档和 Query 按照指定格式弄好
- **优点**：
  - 提升精度，召回的文档中进一步找答案
  - 多源融合：多种不同召回方案的结果可以融合处理
- **缺点**：
  - 增加耗时、消耗算力
  - ReRank Model 有上下文长度限制
  - 对查询扩写不友好：如果系统使用了"多查询（Multi-Query）"技术，生成了 5 个变体问题，那么重排序的计算量会直接乘以 5 倍

### 答案生成

- **可能存在的问题**：
  - 没有搜索到内容，大模型凭空捏造答案
  - 检索到了相关信息，但是大模型没有按照要求生成答案
    - **知识冲突**：多个文档相同内容不一致
    - **丢失中间信息**：LLM 对于超长上下文的开头和结尾比较敏感，若关键答案在中间则可能比较容易被模型忽略
    - **模型忽略参考资料**：模型不基于文档生成
  - 生成的结果不全面，希望进一步优化

- **优化方案**：

  1. **选择适合场景的模型**
     - 简单查询总结，小规模 LLM
     - 需要复杂逻辑推理，需要大规模的 LLM
     - 资料过多，那么上下文空间要足够长的 LLM
     - 特定领域，则可以用特定领域训练的 LLM

  2. **优化 Prompt**
     - **明确要求不编造答案**
       - 可通过强约束指令，在提示词中明确"禁止"模型使用外部知识
       - 可通过强制引用手段，要求模型在句末标注信息来源（答案溯源约束）
     - **添加内容分隔标记**：XML 标记
     - **根据问题类型调整模板**
       - 不同问题的回答范式可能是不同的，你可以借助大模型识别问题类型，然后映射使用不同的提示词模板
       - 比如有些问题，你希望大模型先输出整体框架，然后再输出细节
       - 有些问题你可能希望大模型言简意赅的给出结论

  3. **模型调参**
     - 相同的问题下，在每次模型调用时传入相同的 seed 值
     - 不要总是用重复的句子，你可以适当调高 `presence_penalty` 值
     - 如果你希望查询事实性的内容，可以适当降低 `temperature` 或 `top_p` 值；反之，查询创造性的内容时，可以适当增加它们的值
     - 如果你需要限制字数（如生成摘要、关键词）、控制成本或减少响应时间的场景，可以适当降低 `max_tokens` 的值；但是若 `max_tokens` 过小，可能会导致输出截断，反之，需要生成大段文本时，可以提高它的值

  4. **模型微调**

     > 如果提示词优化到了极限，效果仍不理想，就需要考虑微调。

     - **SFT（指令微调）**：准备一批"问题 - 参考资料 - 标准答案"的语料，对大模型进行微调
     - **训练目标**：专门训练模型学会"根据参考资料回答"以及"当资料不足时学会拒绝回答"

## GraphRAG

通过结合知识图谱（Knowledge Graph, KG）和大语言模型（LLM），解决传统 RAG 在处理复杂问题时的局限性。

它不仅存储文本向量，还通过 NLP 技术从文本块（trunk）中提取实体（Entity）、关系（Relationship）和属性（Property），构建成一张复杂的知识网（图）。在检索时，可将相连的节点（文本块）一起召回。

> 这里的关系可以是跨文本块的关系

### 解决的问题

- **多跳问题**：问题的解决需要多个文本块（trunk）
  - 问题：如果问题是"A 的导师的导师是谁？"。文档1 提到了 A 的导师是 B，文档2 提到了 B 的导师是 C。
  - 传统 RAG 局限：向量检索可能只找回包含"A 的导师"的片段，由于语义距离，不一定能找回"B 的导师"的片段，导致模型无法回答。
  - Graph RAG 优势：通过图路径检索可以找到答案

- **全局理解**
  - 问题：这部小说的主旨是什么？
  - 传统 RAG 局限：它擅长"局部搜索"（寻找具体的某段话），但不擅长总结全书，因为它很难一次性把所有文本块都塞进 Prompt。
  - Graph RAG 优势：通过将图谱划分为不同的层级和簇，对每个簇预先生成摘要

### 工作原理

- **第一阶段：索引构建（如何建图）**
  - 文本切分 → 提取实体和关系 → 将所有三元组构建图 → 用 LLM 为每个簇生成总结

- **第二阶段：查询检索**
  - **局部检索**：定位具体实体 → 提取邻居节点 → 将数据喂给 LLM
  - **全局检索**：检索预先的摘要 → 汇总结论给出全局性回答

## RAG 的评估方案

### Ragas 评估框架

#### 简单介绍

- Ragas（Retrieval Augmented Generation Assessment）是一款广受欢迎的用于 RAG 框架效果评估的自动化工具。它提供了上下文（contexts）维度和答案（answer）维度的效果指标计算，实验显示其指标结果的准确性已经非常接近人工评测结果。
- Ragas 的核心理念是使用 LLM 来评估 LLM（LLM-as-a-Judge），通过一系列自动化的指标来衡量 RAG 系统的性能。

#### 评估指标（Rag Metric）

分成两个维度：**检索（Retrieval）** 和 **生成（Generation）**。

**检索相关指标**：

- **上下文精度（Context Precision）**：评估在针对给定查询（Query）所检索到的上下文中，检索器是否能够将相关的文本块（Chunks）排在比不相关文本块更高的位置
- **上下文召回率（Context Recall）**：衡量的是在所有相关的文档中，有多少被成功检索了出来。核心关注点在于"不遗漏"重要结果

**生成相关指标**：

- **忠实度（Faithfulness）**：用于衡量生成的回答与检索到的上下文在事实逻辑上的一致性。其分值范围为 0 到 1，得分越高说明一致性越好
- **答案相关性（Answer Relevancy）**：用于衡量生成的回答与用户输入之间的相关程度。其分值范围为 0 到 1，得分越高表示回答与用户输入的匹配度越高
- **噪声敏感度（Noise Sensitivity）**：用于衡量系统在利用检索到的文档（无论该文档是相关的还是不相关的）时，因提供错误回答而产生失误的频率。该分数范围在 0 到 1 之间，数值越低表示性能越好

#### 评测集生成（Testset Generation for RAG）

Ragas 提供了一个强大的工具，可以自动生成测试集。
