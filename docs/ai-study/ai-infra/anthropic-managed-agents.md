# Anthropic Managed Agents解析

> 参考资料：
> - Anthropic 技术博客《Scaling Managed Agents: Decoupling the brain from the hands》（2026.04.09，作者：Lance Martin, Gabe Cemaj, Michael Cohen）

## 一、从Claude Code到Managed Agents

### 1.1 Agent的"宠物"困境

搭一个能跑的Agent demo不难，但把它送到生产环境稳定运行，是另一回事——沙盒化的代码执行环境、长时间运行的状态管理、凭证保险箱、容器崩溃恢复、端到端日志追踪……这些纯粹的基础设施工作可能比Agent本身的业务逻辑还重，而且每个团队都在重复造轮子。

单体架构把所有组件塞进一个容器，对原型阶段完全够用。但一旦Agent需要长时间自主运行、规模化部署，Harness（控制循环）、沙盒（执行环境）、会话状态三者紧耦合的弊端就暴露无遗：

| 痛点 | 表现 |
|------|------|
| **稳定性** | 容器崩溃 = 会话全丢，三者绑在一起 |
| **安全性** | LLM与执行环境耦合，提示词注入可直接导致凭证泄露 |
| **性能** | 预分配容器资源浪费严重，按需创建又冷启动太慢 |

在Managed Agents之前，Agent基础设施存在明显的定位空缺：开源框架（LangGraph、CrewAI）解决了编排问题，但运行环境和安全留给开发者；云平台（AWS Bedrock Agents）提供了托管基础设施，但编排与模型缺乏深度整合。**没有哪个方案同时做到了"托管运行时"和"模型提供商级别的深度集成"**——这正是Managed Agents的差异化定位。

这些Agent被当作"宠物"（Pets）来照料——出故障时修复而非替换，配置独特且手工维护，一旦死亡损失不可挽回。

### 1.2 Managed Agents能力全景

2026年4月，Anthropic发布Managed Agents，这是一套可组合的API，用于构建和部署云托管的AI Agent。与市面上的Agent框架不同，Anthropic卖的核心不是框架，而是**Harness（Agent编排引擎）**——一个经过调优的编排循环，自动处理工具调用决策、上下文管理、错误恢复，并且随模型能力升级自动演进。

Managed Agents提供六大核心能力：

| 能力 | 说明 | 成熟度 |
|------|------|--------|
| **生产级沙箱** | 每个Agent跑在安全的云容器里，可预装Python/Node.js/Go等环境，配置网络访问规则，挂载文件 | ✅ 已发布 |
| **长时间运行的Session** | Agent可自主运行数小时，断连自动恢复，进度和产出持久化 | ✅ 已发布 |
| **内置编排Harness** | 经过调优的Agent Harness，自动处理工具调用、上下文管理和错误恢复，内置prompt caching、compaction等优化 | ✅ 已发布 |
| **多Agent协调** | 一个Agent可以启动其他Agent，分配子任务并行处理 | 🔬 研究预览 |
| **自评估能力** | 定义成功标准，Claude自行迭代直到达标 | 🔬 研究预览 |
| **治理工具** | Scoped Permissions、身份管理、执行追踪，Session tracing和集成分析内置在Claude Console | ✅ 已发布 |

开发者可以用**自然语言描述**来定义Agent，也可以用**YAML文件**。同时支持MCP服务器和Agent Skills——这意味着Agent的能力边界不是硬编码的，而是可以通过MCP生态无限扩展。

### 1.3 "解耦"作为破局之道

核心理念一句话概括：

> **将Brain（推理决策）从Hands（执行操作）中解耦出来，用Session（持久化事件日志）作为状态的唯一真相来源。**

这一架构使得每个组件都可以独立伸缩、独立故障恢复、独立演进——Agent从"宠物"变成了"牛群"（Cattle）：只有编号没有名字，出现故障直接替换而非修复，配置完全自动化且可复制。

这个比喻最早由Bill Baker在2012年提出，用以描述服务器管理的两种范式：宠物模式——每台服务器有名字，出故障时修复而非替换，配置独特且手工维护；牛群模式——服务器只有编号，故障时直接替换，配置完全自动化且可复制。云计算的演进史本质上就是这场"宠物到牛群"的迁移——AWS EC2、Kubernetes、Serverless等技术的普及，让"牛群模式"成为现代基础设施的默认选择。Managed Agents把同样的范式带到了Agent基础设施。

## 二、核心架构：Brain-Hands-Session三元解耦

Managed Agents将单体Agent解构为三个独立且松耦合的组件：

![Brain-Hands-Session三元解耦架构](/ai-study/anthropic-managed-agents/three-way-decoupling-mechanism.svg)

各组件通过精确定义的接口进行交互，这些接口是Meta-Harness设计理念的基础——不对具体实现做假设，只对接口形状做假设：

![Managed Agents接口定义](/ai-study/anthropic-managed-agents/managed-agents-interfaces.png)

### 2.1 Brain：推理与决策的中枢

Brain由Claude LLM及其Harness（控制循环）组成，职责三层：

- **推理**：理解用户意图，分析当前状态，制定策略
- **决策**：选择下一步行动——调哪个工具、传什么参数
- **编排**：管理多步骤任务的执行流程，处理异常

Brain的工具调用接口被设计得极简：

```
execute(name, input) → string
```

Brain不需要知道工具怎么实现、在哪运行、凭证怎么获取——它只需要知道工具的名称和输入参数，然后接收字符串形式的返回结果。这种设计体现了关注点分离（Separation of Concerns）的经典原则：Brain专注于它最擅长的事情——推理和决策——而将执行细节完全委托给其他组件。

#### Harness：控制循环的工程实现

Harness是Brain的工程载体，负责实现LLM与外部世界的交互循环：

![Harness控制循环流程](/ai-study/anthropic-managed-agents/harness-loop-mechanism.svg)

一个典型的Harness执行流程如下：

1. 接收用户输入或系统事件
2. 构建包含系统提示、历史上下文、当前输入的完整提示词
3. 调用LLM获取响应
4. 解析响应，识别工具调用请求
5. 通过 `execute` 接口调用相应工具
6. 将工具执行结果反馈给LLM
7. 重复步骤3-6，直到任务完成

Harness的设计空间其实很广阔——Claude Code是一个优秀的Harness实现，但绝不是唯一选择。不同场景需要不同的策略：有的需要更强的自主性，有的需要更严格的人类监督，有的需要特定的安全策略。解耦的价值就在于此：**Brain的推理能力不变，但Harness可以根据场景灵活替换**。

但这里有一个容易被忽略的问题：**Harness编码的是"模型做不到什么"的假设，而这些假设会过时**。一个具体例子：Sonnet 4.5在接近上下文限制时会提前收工（Anthropic内部称之为"context anxiety"），为此他们在Harness里加了上下文重置来应对。但换到Opus 4.5之后，这个行为消失了，重置反而变成了多余的负担。这正是解耦的意义——当模型能力演进时，Harness可以独立更新，而不需要重建整个系统。

### 2.2 Hands：无状态的执行环境

Hands是Agent的"执行肢体"，负责实际运行代码、编辑文件、调用API等操作。在Managed Agents架构中，Hands被实现为**完全无状态的沙盒**。每个沙盒临时创建、用完即弃，带来三个关键优势：

- **弹性伸缩**：沙盒数量随负载动态调整
- **故障隔离**：单个沙盒崩溃不影响其他组件
- **安全边界**：天然隔离层，限制攻击影响范围

沙盒与Brain之间通过**MCP（Model Context Protocol）协议**通信。MCP是Anthropic于2024年11月开源的协议标准，现已成为AI Agent领域的事实标准。敏感凭证不存储在沙盒中，而是由独立的MCP代理负责获取和管理——沙盒中的代码只能通过MCP代理发起请求，无法直接接触令牌。

具体的安全实践值得展开：**Git令牌在初始化时写入本地remote，Agent不经手**——沙盒中的代码可以执行git push/pull，但令牌已经预置在git配置中，Agent代码从不直接读写令牌。**OAuth令牌存储在安全保管库（Vault）中，通过MCP代理调用**——当Agent需要访问外部API时，MCP代理从Vault获取令牌、发起请求、返回结果，令牌本身从不暴露给沙盒。这种"凭证永不进入执行环境"的设计，使得安全边界因解耦而变得干净利落——在耦合架构中，Claude生成的不可信代码和敏感凭证共处同一个容器，提示词注入可以直接窃取凭证；解耦后，凭证永远不在沙盒里，攻击面被大幅收窄。

沙盒基于容器技术实现，提供企业级的资源管理能力：

- **资源配额**：每个沙盒可设置CPU、内存、磁盘使用上限
- **网络隔离**：沙盒的网络访问可以被精细控制，防止数据外泄
- **执行超时**：长时间运行的操作可被自动终止，防止资源耗尽
- **审计日志**：所有执行操作都被记录，满足合规要求

### 2.3 Session：从"宠物"到"牛群"的关键使能器

Session是最容易被低估但最重要的创新。Anthropic在博客中有一句值得反复琢磨的话：

> **"The session is not Claude's context window."**

这句话点明了Session的本质——它是一个**追加式（Append-only）的持久化事件日志**，记录Agent的所有交互，而非LLM的临时上下文。追加式意味着事件只能写入、不能修改或删除，这保证了历史记录的完整性和不可篡改性。

这里有一个关键区分：**Context Window是临时的，Session是持久的**。Context Window是一次模型调用能看到的token，它会变贵、退化、被压缩、被无关信息污染；而Session是独立于模型调用之外的事件流，持久存储于Harness之外。正是因为这个分离，Harness崩了才可以用`wake(sessionId)`重启，新的Harness调用`getSession(id)`拉取事件流，即可从断点继续执行。

每个事件包含四个要素：

| 要素 | 说明 |
|------|------|
| 事件类型 | 用户输入、LLM响应、工具调用、工具结果、系统事件等 |
| 时间戳 | 精确到毫秒的事件发生时间 |
| 内容详情 | 提示词文本、工具参数、执行结果等具体内容 |
| 元数据 | 关联的会话ID、用户ID、版本信息等 |

核心接口：

```
getEvents(filter) → Event[]
```

这个接口看似简单，却蕴含着深刻的架构智慧——它让Harness具备了故障恢复能力：如果Harness进程崩溃，新的Harness实例可以通过 `getEvents()` 获取完整的历史上下文，无缝接管会话。用户几乎感知不到中断。

- **可恢复性**：Harness崩溃后，新实例可通过 `getEvents()` 获取完整历史，无缝接管
- **可观测性**：所有操作有迹可循
- **可查询性**：支持按类型、时间范围等检索历史

#### 线性日志的局限与扩展空间

`getEvents()` 支持按位置切片、回退、按时间点回溯，相比将所有内容塞进Context Window已有明显进步。但在高步数场景下，它存在一个实际短板：**线性日志擅长定位断点，却不擅长语义检索**。

举个例子：Agent执行200多步后突然出现依赖冲突，需要回溯历史查找是否处理过类似冲突。用`getEvents()`按位置切片检索，本质仍是全文遍历，既耗时、成本高，准确性也无法保证。

好在Anthropic的设计预留了良好的扩展空间——Harness可以在把事件传给模型之前做任意变换，Context Engineering的细节由Harness承担，Session仅负责保证数据的持久性和可查询性。这意味着我们可以在Session之上叠加语义检索层（如向量数据库），让记忆从"线性"升级为"立体"，而无需改动底层架构。

#### Event Sourcing视角：Session是事件日志，Context Window是投影

从分布式系统的角度看，Managed Agents与Event Sourcing（事件溯源）模式有很强的同构关系：**session是append-only event log（事实来源），context window是从事件流投影出的临时视图**。

这解释了为什么context compaction（上下文压缩）只是projection策略——当Context Window快满时，Harness对事件进行摘要、截断，但这只是投影出一个适合当前窗口的视图，原始事件仍完整保留在session中。就像数据库的物化视图可以刷新重建，但底层表数据不能丢。"上下文被压缩"不等于"历史被删除"。

与传统durable workflow（如Temporal）的关键差异在于：传统workflow要求replay确定性，而LLM推理天然是概率性的。因此恢复时不是重跑过去的推理，而是重放已固化的历史事实，只在新的决策边界继续调用模型。

Session的存在使得Harness从"宠物"变为"牛群"：

| | 宠物模式 | 牛群模式 |
|---|---------|---------|
| Harness | 与会话绑定，死亡 = 会话丢失 | 无状态可互换，任何实例可服务任何会话 |
| 伸缩 | 无法水平扩展 | 动态增减实例 |
| 故障 | 不可恢复 | 自动迁移会话 |
| 部署 | 需停机 | 零停机滚动更新 |

### 2.4 协同工作流示例

以"分析过去一周的销售数据并生成报告"为例：

1. **初始化**：创建Session，分配Harness实例
2. **输入处理**：用户输入记录到Session，Harness构建提示词调用Brain
3. **推理**：Brain确定需要调用 `query_database` → `analyze_data` → `save_report`
4. **工具调用**：每一步，Harness创建临时沙盒，MCP代理获取凭证执行，结果返回Brain
5. **迭代优化**：Brain根据中间结果调整策略
6. **结束**：所有事件持久化到Session，沙盒销毁，Harness释放

## 三、为什么是"三元"解耦？架构取舍分析

Managed Agents将单体Agent拆成Brain、Hands、Session三个组件，而不是两个或四个。这个"三"不是随意定的，而是对Agent本质的重新理解——Agent不是一个程序，而是一个由多个专业化组件协作完成的分布式系统。

### 3.1 为什么不是二元？

如果只拆成"推理"和"执行"两部分，状态该放在哪？

- 放在Brain里？Brain变成了有状态的，无法随意替换和扩展
- 放在Hands里？执行环境既要管运行又要管状态，职责混乱
- 两边都放？没有统一真相来源，故障恢复和状态同步变成噩梦

**状态必须独立出来**，这是从分布式系统实践中得出的铁律——无状态服务才能水平扩展，持久化状态必须有唯一真相来源。

### 3.2 为什么不是四元或更多？

进一步拆分理论上可行，但会带来新的问题：

- **通信开销**：组件越多，跨组件调用越频繁，延迟和复杂度随之增加
- **运维负担**：每个独立组件都需要部署、监控、故障恢复
- **心智成本**：开发者需要理解更多组件间的交互关系

三元解耦在"职责清晰"和"复杂度可控"之间找到了一个不错的平衡点。Brain管推理、Hands管执行、Session管状态——各司其职，边界清晰。

### 3.3 Agent基础设施为何深陷"宠物"困境

![宠物模式vs牛群模式对比](/ai-study/anthropic-managed-agents/pets-vs-cattle-comparison.svg)

在Managed Agents之前，大多数Agent架构都面临三重绑定：

**困境一：Harness与会话绑定**。Harness不仅负责与LLM交互，还维护对话历史、管理工具状态、跟踪任务进度。结果是：Harness崩溃 = 会话丢失，Harness升级 = 会话中断，Harness迁移 = 状态同步噩梦。

**困境二：沙盒与Harness绑定**。执行环境与Harness绑定在一起，沙盒中可能运行着长时间任务、缓存着中间结果。结果是：沙盒资源无法独立伸缩，沙盒故障影响整个Harness，沙盒升级需要重启Harness。

**困境三：状态管理混乱**。状态散落在各个组件中——Harness中有对话历史，沙盒中有执行状态，外部数据库中有任务元数据。没有统一的真相来源（Single Source of Truth），故障恢复和状态同步变得异常复杂。

### 3.4 解耦架构如何实现"牛群模式"

Managed Agents通过三元解耦，逐条打破了这三重困境：

**Harness无状态化**：Harness不维护任何会话专属状态，所有状态通过 `getEvents()` 从Session获取。这意味着任何Harness实例都可以服务任何会话，Harness可以随意创建、销毁、重启，升级对用户完全透明，可以水平扩展应对高并发。

**沙盒完全独立**：沙盒与Harness之间通过MCP协议通信，两者是完全独立的进程。沙盒按需创建、用完即弃，故障不影响Harness或其他沙盒，资源使用可以被精确控制和计费，可以运行在不同的基础设施上（本地、云端、边缘）。

**Session作为唯一真相来源**：Session是系统中唯一持久化的状态存储，提供完整的可恢复性（任何组件故障都可从Session重建）、强大的可观测性（所有操作有迹可循）、灵活的查询能力和清晰的状态边界。

### 3.5 Meta-Harness：为"尚未想到的程序"设计系统

Managed Agents的设计目标不只是解决眼前的问题，而是为未来留出空间。Anthropic工程博客把这个架构叫"meta-harness"——不对具体的Harness实现做假设，只对Claude需要的接口形状做假设。

这个思路借鉴了操作系统的设计哲学。操作系统通过虚拟化硬件（进程、文件）来支持还不存在的程序，因为**抽象比硬件活得久**。同样地，Managed Agents通过虚拟化Agent的组成部分（Session、Harness、Sandbox）来支持尚未设想的Agent应用。

每双手就是一个工具调用：`execute(name, input) → string`。Harness不知道沙箱是容器、手机还是Pokémon模拟器——它只关心接口形状。因为手不绑定大脑，大脑之间可以互相传递手。这种接口先行、实现后置的设计，是让系统随模型能力持续演进的关键。

### 3.6 "牛群模式"的运营优势

| 优势 | 说明 |
|------|------|
| **高可用** | 单点故障不影响整体，组件故障可快速自动恢复，支持零停机滚动更新 |
| **弹性伸缩** | 根据负载动态调整实例数，无需预分配资源，快速响应流量峰值 |
| **成本优化** | 按实际使用付费，沙盒短暂生命周期意味着更低资源占用，精细配额防止成本失控 |
| **安全合规** | 沙盒隔离降低攻击面，完整审计日志满足合规，凭证与执行环境分离 |

## 四、性能与安全：量化收益与架构保障

### 4.1 性能提升

Anthropic公布的数据：

| 指标 | 改进幅度 |
|------|---------|
| p50 TTFT | 降低约60% |
| p95 TTFT | 降低超过90% |

TTFT（Time To First Token，首Token时间）是衡量LLM应用响应速度的关键指标。在传统预分配容器架构中，TTFT包含了容器启动时间、环境初始化时间、依赖加载时间等固定开销。Managed Agents通过以下机制优化：

- **按需创建沙盒**：容器仅在工具调用需要时才创建，而非预先分配
- **并行化初始化**：Harness可以在等待LLM响应的同时准备沙盒资源
- **资源预热池**：维护一个轻量级的沙盒池，进一步降低冷启动时间

资源利用率方面：CPU从预分配架构的15-20%提升到60%以上，基础设施成本降低50-70%。

### 4.2 安全架构：纵深防御

![纵深防御架构](/ai-study/anthropic-managed-agents/defense-in-depth-architecture.svg)

Managed Agents的安全架构采用**纵深防御**理念——多层防护，任一层被突破，攻击者也无法达到最终目标：

**第一层：沙盒隔离**

进程隔离、文件系统隔离、网络白名单、资源硬上限。

**第二层：MCP代理的凭证管理（核心创新）**

```
传统架构：  沙盒 ← 包含凭证 → 直接调用外部API    ❌ 凭证可被窃取

Managed Agents：  沙盒 → MCP代理（获取凭证）→ 执行调用
                         ↑
                    安全Vault（存储凭证）       ✅ 凭证从不进入沙盒
```

即使攻击者成功诱导Agent执行恶意代码，代码也只能通过MCP代理发起请求，而无法直接获取凭证本身。这从根本上消除了提示词注入攻击导致凭证泄露的可能性。值得注意的是，**这种安全优势是解耦架构的直接产物**——在耦合架构中，不可信代码与凭证共处同一容器，安全边界模糊不清；解耦让安全边界变得清晰：沙盒只管执行，凭证只管授权，两者永不相见。

**第三层：Session审计能力**

完整事件日志提供：操作可追溯、异常可检测、事件可响应、合规可证明。

**第四层：Harness安全策略**

工具白名单、输入验证、输出过滤、关键操作人工审核。

### 4.3 安全性对比

| 攻击向量 | 传统架构风险 | Managed Agents缓解 |
|---------|------------|-------------------|
| 提示词注入 | 高：可能导致凭证泄露 | 低：凭证从不进入沙盒 |
| 代码执行 | 高：可能获取系统权限 | 中：沙盒隔离限制影响 |
| 数据外泄 | 中：可能通过网络泄露 | 低：网络访问可精细控制 |
| 拒绝服务 | 中：可能耗尽资源 | 低：资源配额限制 |

## 五、当前局限与演进方向

### 5.1 已具备的治理能力

在讨论局限之前，先看Managed Agents已经提供的治理能力：

- **作用域权限（Scoped Permissions）**：Agent的每个工具调用都受权限约束，避免过度授权
- **身份管理**：Agent以明确的身份运行，操作可追溯到具体的Agent实例和租户
- **执行追踪（Session Tracing）**：内置在Claude Console中，可以检查每一个工具调用、决策点和失败模式
- **集成分析**：直接在控制台查看Agent行为统计和异常检测

此外，Anthropic正在研究预览阶段的**自评估能力**：用户定义成功标准，Claude自己迭代直到达标。在内部测试中，结构化文件生成任务的成功率比标准prompting方式提高了最多10个百分点，且提升在难题上最明显。这暗示了Harness的下一个演进方向——从"执行者"向"自省者"进化。

### 5.2 四大局限

**局限一：Session的语义检索能力不足**

当前Session是基于事件的时间序列日志，`getEvents()` 擅长按位置回溯，却不擅长语义检索。当事件达到数百条时，Agent需要查找"是否遇到过类似问题"，线性遍历的成本和准确性都无法满足需求——顺序读取和语义检索需要不同的数据结构。

**局限二：多Brain记忆孤岛**

解耦架构让算力弹性得到极大提升，但代价是每个Session相互独立，并发的Brain之间完全不知道对方在做什么。Brain A花了40分钟解决了棘手的SQL注入问题，Brain B遇到同样的问题时仍需从头来过——智能变成了一次性消耗品。这就是stateless brain的代价：容易scale，但无法通过经验积累变得聪明。

**局限三：多Agent协作缺失**

架构主要面向单Agent。虽然Anthropic已推出多Agent协调的研究预览，但通信协议、任务分解、冲突解决、共享状态管理等核心问题尚未成熟。目前的多Agent协调更像是"主从委派"，而非真正的对等协作。

**局限四：长期记忆不足**

Session面向短期会话，跨会话的知识积累和个性化能力需要扩展。

### 5.3 演进方向

#### 方向一：给Session接入语义记忆层

基于Harness的扩展性设计，可以在Session之上叠加向量数据库，实现语义检索。核心思路：在事件写入Session的同时，对关键决策类事件（如`decision`、`strategy`、`resolution`、`error_handling`）进行嵌入并写入向量存储。当Agent需要回溯历史时，无需遍历事件流，只需发起语义查询即可。

三个工程细节需要注意：
- **事件筛选**：并非所有事件都需嵌入，工具调用中间状态、重试日志等噪声事件会稀释检索质量
- **一致性边界**：Harness故障时可能出现Session事件已写入但向量存储未完成的情况，需设计补写机制
- **成本控制**：批量嵌入+异步队列比每步实时嵌入更具实用性

#### 方向二：给Many Brains建立共享经验池

解决记忆孤岛的自然思路是让所有Brain共享一个向量记忆层。每个Agent Session结束后，将关键决策、遇到的问题、用过的解法嵌入后写进共享Collection。下一个Brain启动时，Harness在初始化阶段做一次语义检索，把相关经验注入上下文。

这个方案面临两个挑战：
- **数据隔离**：多租户场景下，A租户的经验不应被B租户检索到。可通过向量数据库的分区键（如`tenant_id`）实现自动隔离
- **记忆质量管理**：错误的解法、过期的经验会被传播放大。需要引入可信度评分（记录成功验证次数）、时间衰减权重和人工抽查机制

#### 方向三：从同步到异步——Agent Runtime的深层演进

![同步vs异步Agent Runtime演进](/ai-study/anthropic-managed-agents/sync-to-async-runtime-evolution.svg)

前面两个方向主要围绕"能力增强"。但顺着Event Sourcing的视角再往前走一步，Managed Agents的架构还隐含着一个更深层的演进方向：**从同步请求-响应模型，演进为原生异步的分布式Agent Runtime。**

当前的Agent交互模型本质上还是同步的：

```
用户请求 → LLM推理 → 同步调用工具 → 等待返回 → 再次推理 → 返回结果
```

但一旦Agent Runtime建立在event log、stateless harness和可替换hands之上，它天然具备演进为异步系统的基础：

```
事件到达 → harness消费session → 生成下一步命令 → 异步派发给tool/sandbox/LLM
→ 外部结果回流为新事件 → 任意worker继续推进session
```

在这种设计下，session是durable state，harness是无状态调度者，而tool调用乃至LLM inference本身，都可以被视作**异步activity**。这带来几个本质性的变化：

- **Agent生命周期不再绑定同步请求**：session持续存在，worker可以随时被唤醒处理新事件，当前worker消失后另一个worker仍可继续推进同一个session。Agent从"同步请求处理器"变成"可持续推进的异步执行体"
- **Tool call不必是同步函数调用**：harness发出command，外部系统异步执行，结果作为event写回session，任意worker继续决策。无论工具是浏览器自动化、代码执行、人工review还是webhook回调，都可统一成异步事件驱动模式
- **LLM inference也可以被activity化**：推理同样可以被建模为一种activity，让系统更容易做推理队列化、优先级调度、超时与取消、多模型路由和弹性扩缩容
- **Interruptibility成为一等能力**：pause/resume/cancel/retry/handoff不是额外贴上的功能，而是架构自然产物。human-in-the-loop也只是"向session注入新事件"的另一种来源

这为规模化Agentic Service提供了真正的分布式基础。但同时，这也意味着要用分布式系统的标准来设计Agent——幂等性（idempotency）、event delivery的at-least-once语义、并发worker推进同一session的冲突处理等问题都需要认真对待。Agent Platform最终会越来越像**workflow engine、actor system和event-driven orchestration的结合体**，而不再只是"会调工具的大模型应用"。

#### 其他方向
- **多Agent编排**：定义Agent间协作协议，开发编排语言和框架
- **自适应Harness**：根据任务自动选择策略，从经验中优化提示词
- **边缘部署**：轻量级Harness和沙盒，支持边缘-云端协同

### 5.4 核心启示

Anthropic的Managed Agents解决了一个核心问题：让Brain和Hands都变成Cattle，任何一个挂掉都不影响任务继续跑——这是可靠性，解得很好。

但可靠性之外，Agent能否从经验中成长才是更关键的问题。人类工程师通过长期工作积累模式识别能力，无需每次从零推理，而当前的Agent却做不到——每个Session启动时，它都是一张白纸。

把Session接上语义记忆，把多个Brain的经验汇入共享向量池，是在给Agent一个可以积累的过去。但这个过去需要被管理：清理错误记忆、更新过期经验、控制跨租户的数据边界——这是需要长期投入的系统工程，不是一次性配置就能完成的。

## 六、关键洞察总结

| # | 洞察 | 核心要点 |
|---|------|---------|
| 1 | 解耦是必然趋势 | Brain/Hands/Session各司其职，独立演进 |
| 2 | Session ≠ Context Window | Session是持久事件流，Context Window是临时token窗口，本质区分是"宠物变牛群"的基石 |
| 3 | 线性日志不擅长语义检索 | 高步数场景下需叠加向量数据库，让记忆从"线性"升级为"立体" |
| 4 | 多Brain的代价是记忆孤岛 | Stateless brain容易scale，但无法通过经验积累变聪明，需共享经验池 |
| 5 | 凭证与执行环境分离是安全核心 | 纵深防御下，即使沙盒被攻破，凭证仍然安全 |
| 6 | 可靠性与经验积累并重 | Cattle化解决了可靠性，但Agent能否从经验中成长才是核心命题 |
| 7 | Harness编码的假设会过时 | Meta-Harness的接口先行设计让系统随模型能力自动演进 |
| 8 | Agent Platform的终局形态 | Workflow engine + actor system + event-driven orchestration的融合体 |
| 9 | Harness工程是新兴学科 | 与传统软件工程有本质差异，评估替代测试、提示词+模型版本是核心变化 |

---

Managed Agents的架构设计，本质上是在回答三个问题：

1. **如何让Agent可靠运行？** 经典的分布式系统原则——解耦、无状态、持久化日志、故障隔离
2. **如何让Agent从经验中成长？** 叠加语义记忆层和共享经验池——前者让回溯从线性遍历变为语义检索，后者让经验从一次性消耗品变为可复用的知识资产
3. **如何让Agent平台走向规模化？** 从同步请求-响应演进为原生异步的分布式Agent Runtime，让pause/resume/cancel/handoff成为架构自然产物

**可靠性是基础，经验积累是进阶，异步分布式是终局。**

### 给不同读者的行动建议

**技术决策者**：评估当前Agent基础设施的成熟度，判断是否需要向解耦架构迁移；投资Harness工程能力建设，建立Agent安全和治理框架。

**软件工程师**：学习Harness工程和提示词工程，尝试Claude Code等Agent工具；为项目创建CLAUDE.md，从Level 1起步逐步提升成熟度。

**产品经理**：识别可以Agent化的工作流程，设计人类在回路的交互模式；建立Agent成功的度量指标，规划能力的渐进式开放。

正如Claude Code创作者Boris Cherny所言：**AI Agent将重塑几乎所有基于计算机的工作**。编程的本质将从"写代码"转向"指导AI写代码"，技能要求将从语法细节转向架构设计，从调试技巧转向评估能力。开发者需要拥抱变化、持续学习，同时保持批判性思维——AI会犯错，人类需要保持判断能力。
