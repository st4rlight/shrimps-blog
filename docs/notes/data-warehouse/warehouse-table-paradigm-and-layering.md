# 数仓表建设范式与分层方案

> 数仓建模是数据仓库建设的核心环节，合理的表结构设计和分层策略直接决定了数据仓库的可用性、可维护性和查询性能。

## 一、数仓表的两种核心建模范式

在数据仓库建设中，最主流且被广泛公认的两种核心建模范式是 **维度建模（Dimensional Modeling）** 和 **范式建模（Normalized Modeling / 3NF）**。它们代表了两种截然不同的设计哲学：前者以查询效率和使用便捷为导向，后者以数据一致性和存储优化为导向。

![维度建模与范式建模范式对比](/notes/data-warehouse/warehouse-table-paradigm-and-layering/dimensional-vs-normalized.svg)

### 1.1 维度建模（Dimensional Modeling）

维度建模由 Ralph Kimball 提出，是数仓建设中**应用最广泛**的建模范式。其核心思想是：**围绕业务过程，用事实表和维度表来组织数据**，以直观的结构支撑高效的多维分析。

**核心概念：**

- **事实表（Fact Table）**：记录业务过程发生的事实，包含度量值（如金额、数量）和关联维度表的外键
- **维度表（Dimension Table）**：描述业务的上下文环境，提供分析视角（如时间、用户、商品、地区）

**设计原则：**

1. **选择业务过程**：确定要分析的业务活动（如下单、支付、发货）
2. **声明粒度**：明确事实表中每一行代表什么（如每笔订单一个明细行）
3. **确定维度**：选择从哪些角度分析业务过程（如时间、用户、商品）
4. **确定事实**：选择需要度量的指标（如订单金额、商品数量）

**优点：**

- **查询性能好**：结构扁平，JOIN 少，适合大规模分析查询
- **易于理解**：业务人员和开发人员都能快速上手，表结构直观
- **开发效率高**：建模逻辑清晰，ETL 开发和维护成本低
- **天然适配 OLAP**：事实表 + 维度表的结构完美匹配多维分析场景

**缺点：**

- **数据冗余**：维度表中允许存在冗余字段（反范式设计），存储空间占用较大
- **数据一致性风险**：冗余数据在维度变更时需同步更新，可能出现不一致
- **灵活性受限**：维度建模围绕预设的业务过程展开，业务变化时可能需要重构

**适用场景：**

- 以分析查询和报表为核心的 OLAP 场景
- 业务分析人员自助取数的场景
- 互联网公司离线数仓的主流选择

#### 1.1.1 星型模型（Star Schema）—— 维度建模的常用实现

星型模型是维度建模中最简单、最常用的实现方式：

- **一个事实表** 居于中心，包含业务度量和外键
- **多个维度表** 直接围绕事实表，每个维度表通过主键与事实表的外键关联
- **维度表不做规范化**，允许存在数据冗余（如省份、城市、区域放在同一个地区维度表中）
- 整体结构呈**星型**，事实表在中心，维度表在四周

![星型模型结构示意](/notes/data-warehouse/warehouse-table-paradigm-and-layering/star-schema-overview.svg)

**核心优势：** 维度表扁平、不规范化，查询通常只需一次 JOIN，性能最优。绝大多数数仓团队的首选方案。

#### 1.1.2 雪花模型（Snowflake Schema）—— 维度建模的规范化实现

雪花模型是维度建模的另一种实现方式，在维度表上进一步做了规范化：

- **事实表** 依然居于中心
- **维度表被拆分**，高级维度表通过外键关联低级维度表
- 整体结构呈**雪花状**，维度存在多层关联

![雪花模型结构示意](/notes/data-warehouse/warehouse-table-paradigm-and-layering/snowflake-schema-overview.svg)

**核心特点：** 维度表规范化，消除冗余，数据一致性好，但多级 JOIN 导致查询性能较差。通常只在个别超大维度（如用户地区层级）上局部采用。

#### 1.1.3 星型模型 vs 雪花模型

| 对比维度 | 星型模型 | 雪花模型 |
|---------|---------|---------|
| 维度表规范化 | 否（反范式） | 是（范式化） |
| 数据冗余 | 多 | 少 |
| 查询性能 | 高（JOIN 少） | 低（JOIN 多） |
| 存储空间 | 大 | 小 |
| 开发复杂度 | 低 | 高 |
| 数据一致性 | 一般（需同步冗余） | 好（单点维护） |

> **实践建议：** 在维度建模的实践中，**绝大多数团队选择星型模型**，原因是查询性能和开发效率在数仓场景中通常比存储节省更重要。更常见的做法是**混合使用**：核心维度用星型模型保持查询性能，对个别超大维度局部采用雪花化拆分。

### 1.2 范式建模（Normalized Modeling / 3NF）

范式建模由 Bill Inmon（数仓之父）倡导，其核心思想是：**按照第三范式（3NF）组织数据，消除数据冗余，保证数据的一致性和完整性**。

**核心概念：**

- **第三范式（3NF）**：每个非主属性既不部分依赖于候选键，也不传递依赖于候选键
- 实质是**消除冗余**和**消除传递依赖**，每个事实只存储一次
- 数据按照**实体-关系（E-R）模型**组织，类似传统关系型数据库的设计方式

**设计原则：**

1. 按照 3NF 规范拆分实体和关系
2. 每个实体独立建表，通过外键关联
3. 消除所有非主属性对主键的传递依赖
4. 追求"单一事实存放点"，数据不冗余

**优点：**

- **数据冗余极小**：每个事实只存储一次，存储空间利用率高
- **数据一致性强**：数据更新只需修改一处，不存在不一致风险
- **扩展性好**：新增实体和关系不影响已有结构
- **适合数据集成**：多源数据整合时，3NF 结构便于维护统一口径

**缺点：**

- **查询性能差**：高度规范化意味着大量 JOIN，分析查询效率极低
- **使用门槛高**：表数量多、关系复杂，业务人员难以理解和使用
- **开发维护成本高**：ETL 逻辑复杂，数据加工链路长
- **不直接适配 OLAP**：需要额外的数据集市层来支撑分析需求

**适用场景：**

- 企业级数据仓库的底层基础层（EDW）
- 对数据一致性和完整性要求极高的场景
- 数据来源复杂、需要统一标准化集成的场景
- 作为维度建模的数据源，先 3NF 再维度化

### 1.3 两种建模范式的对比与选择

| 对比维度 | 维度建模 | 范式建模（3NF） |
|---------|---------|----------------|
| 提出者 | Ralph Kimball | Bill Inmon |
| 核心思想 | 围绕业务过程，事实+维度 | 按第三范式消除冗余 |
| 数据冗余 | 允许冗余（反范式） | 消除冗余（范式化） |
| 查询性能 | 高（JOIN 少） | 低（JOIN 多） |
| 使用便捷性 | 高，直观易用 | 低，关系复杂 |
| 开发效率 | 高 | 低 |
| 数据一致性 | 一般（冗余需同步） | 强（单点维护） |
| 扩展性 | 围绕业务过程，变更需重构 | 实体独立，扩展灵活 |
| 典型应用 | 数据集市、OLAP 分析 | 企业级数据仓库基础层 |

**业界实践：Kimball 与 Inmon 的融合**

> 在实际数仓建设中，纯粹的 Kimball 方案和纯粹的 Inmon 方案都很少见。**业界主流做法是两者融合**：
>
> - **Inmon 的部分**：在数仓的底层（ODS → DWD），采用偏向 3NF 的设计，确保数据来源标准化、一致性有保障
> - **Kimball 的部分**：在数仓的上层（DWS → ADS），采用维度建模，确保查询性能和使用便捷
>
> 这种融合方式既保证了数据的"单一事实存放点"，又保证了分析的"高效可用"，是目前最成熟的数仓建模实践。

## 二、数仓表的分层方案

数仓分层是数据仓库架构设计的核心原则。合理的分层可以解耦数据生产流程、降低重复计算、保证数据质量，并提升数据的复用性和可维护性。

### 2.1 为什么要分层

在数仓建设中不分层或分层不合理，会导致以下问题：

- **数据口径不一致**：同一指标在不同报表中数值不同，无法对齐
- **重复开发严重**：多个下游任务重复清洗相同数据，浪费资源
- **问题排查困难**：数据出问题时，无法快速定位是哪个环节出错
- **修改牵一发动全身**：上游表结构变更，下游所有任务都可能受影响
- **数据治理混乱**：没有清晰的边界，数据质量无法保障

分层的核心目标就是：**通过明确的层次划分，让每一层各司其职，实现数据的有序流转和治理。**

![数仓数据流转机制](/notes/data-warehouse/warehouse-table-paradigm-and-layering/warehouse-layering-mechanism.svg)

### 2.2 业界主流分层架构

目前业界最广泛采用的分层方案是 **ODS → DWD → DWS → ADS** 四层架构，部分团队会根据需要增加 DIM 层。

![离线数仓分层架构总览](/notes/data-warehouse/warehouse-table-paradigm-and-layering/warehouse-layering-overview.svg)

### 2.3 各层详细说明

#### 2.3.1 ODS 层（Operational Data Store，贴源层）

**定位：** 原始数据的落地层，与数据源保持一致。

**职责：**

- 从业务数据库、日志系统、第三方数据源等**原样同步**数据
- **不做任何清洗和转换**，保持与源系统完全一致
- 增加必要的**分区字段**（如 dt）和**ETL 元数据**（如导入时间）
- 作为数仓的**数据底座**，后续所有层次的数据均来源于此

**设计原则：**

- 表名规范：`ods_{业务线/数据源}_{原始表名}_{增量/全量标识}`
- 通常按天分区存储，增量表加 `_inc` 后缀，全量表加 `_full` 后缀
- 保留历史全量快照，支持数据回溯

**示例：**

| 表名 | 说明 |
|------|------|
| `ods_trade_order_full` | 订单全量快照表 |
| `ods_trade_order_detail_inc` | 订单明细增量表 |
| `ods_user_log_inc` | 用户行为日志增量表 |

#### 2.3.2 DWD 层（Data Warehouse Detail，明细层）

**定位：** 标准化、清洗后的明细事实数据层，是数仓中最核心的层。

**职责：**

- **数据清洗**：去除脏数据、空值处理、异常值过滤
- **数据标准化**：统一字段命名、数据类型、编码规范
- **维度退化**：将常用维度字段冗余到事实表中，减少 JOIN
- **多源合并**：将相同业务过程的多源数据合并为统一明细表
- **建立事实表**：围绕业务过程，构建明细粒度的事实表

**设计原则：**

- 表名规范：`dwd_{业务线}_{业务过程}_{粒度}_{增量/全量标识}`
- 以**业务过程**为驱动建模，一个业务过程对应一张明细事实表
- 明细层保持最细粒度，不做聚合

**常见事实表类型：**

| 类型 | 说明 | 示例 |
|------|------|------|
| 事务事实表 | 记录业务事务的发生 | 下单、支付、退款 |
| 周期快照事实表 | 按固定周期记录状态 | 每日库存快照 |
| 累积快照事实表 | 记录业务全生命周期的多个里程碑 | 订单从下单到签收 |

**示例：**

| 表名 | 说明 |
|------|------|
| `dwd_trade_order_detail_inc` | 交易域下单明细事实表 |
| `dwd_trade_pay_detail_inc` | 交易域支付明细事实表 |
| `dwd_user_login_inc` | 用户域登录明细事实表 |

#### 2.3.3 DIM 层（Dimension，维度层）

> DIM 层并非所有数仓架构的独立层，但在维度建模实践中，将公共维度表独立出来管理是非常有价值的。

**定位：** 公共维度表的统一管理层。

**职责：**

- 管理**所有公共维度表**，如用户维度、商品维度、时间维度、地区维度等
- 提供**一致的维度属性**，避免各团队重复建设
- 维护维度的**历史变更**（SCD 缓慢变化维处理）
- 支持**维度退化**：将高频使用的维度属性冗余到 DWD 事实表中

**设计原则：**

- 表名规范：`dim_{业务线}_{维度名称}_{全量/拉链表标识}`
- 维度表通常为全量快照或拉链表
- 高基数维度（如用户维度）和低基数维度（如地区维度）分开管理

**缓慢变化维（SCD）处理策略：**

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| SCD1（覆盖） | 直接覆盖旧值 | 不需要保留历史 |
| SCD2（新增行） | 新增一行记录，标记有效期 | 需要完整历史追踪 |
| SCD3（新增列） | 增加列记录旧值 | 只需记录最近一次变更 |

**示例：**

| 表名 | 说明 |
|------|------|
| `dim_user_full` | 用户维度全量快照表 |
| `dim_sku_full` | 商品 SKU 维度全量表 |
| `dim_date_full` | 日期维度全量表 |

#### 2.3.4 DWS 层（Data Warehouse Summary，汇总层）

**定位：** 面向分析主题的汇总数据层，构建公共汇总指标。

**职责：**

- 以**主题域**为驱动，对 DWD 层明细数据进行**轻度或重度聚合**
- 构建公共**汇总宽表**，将同一主题下的多个业务过程指标合并
- 产出**可复用的公共指标**，避免下游重复计算
- 支持**多维度自由下钻和上卷**

**设计原则：**

- 表名规范：`dws_{业务线}_{主题域}_{汇总粒度}_{增量/全量标识}`
- 汇总粒度要明确：如用户日汇总、商品日汇总、店铺月汇总等
- 指标命名要规范：如 `pay_amt_1d`（近1天支付金额）、`order_cnt_7d`（近7天下单次数）

**常见汇总粒度：**

| 粒度 | 说明 | 示例 |
|------|------|------|
| 1d | 近1天汇总 | `dws_trade_user_1d` |
| nd | 近N天汇总 | `dws_trade_user_nd` |
| td | 历史截至当天累计 | `dws_trade_user_td` |

**示例：**

| 表名 | 说明 |
|------|------|
| `dws_trade_user_1d` | 交易域用户粒度1日汇总表 |
| `dws_trade_user_nd` | 交易域用户粒度N日汇总表 |
| `dws_trade_sku_1d` | 交易域商品粒度1日汇总表 |

#### 2.3.5 ADS 层（Application Data Store，应用层）

**定位：** 面向具体应用场景的数据服务层，直接对接下游消费方。

**职责：**

- 根据报表、接口、服务等**具体应用需求**组织数据
- 产出高度聚合的**结果表**，直接支撑 BI 展示和数据服务
- 可直接对接 MySQL、ClickHouse、Redis 等应用数据库
- 通常不再被其他数仓任务引用，是数据流转的**终点**

**设计原则：**

- 表名规范：`ads_{业务线}_{应用场景}_{其他标识}`
- 面向需求驱动，有什么需求就建什么表
- 数据量小、查询要求高，可直接同步到应用数据库

**示例：**

| 表名 | 说明 |
|------|------|
| `ads_user_retention` | 用户留存分析表 |
| `ads_gmv_stats` | GMV 统计报表 |
| `ads_top_selling_sku` | 热销商品排行表 |

### 2.4 数据流转示意

![数仓数据流转全貌](/notes/data-warehouse/warehouse-table-paradigm-and-layering/warehouse-data-flow.svg)

**核心流转规则：**

1. 数据只能**从上游流向下游**，严禁反向依赖
2. DWD 层是核心，所有事实数据必须经过 DWD 标准化
3. DWS 层只依赖 DWD + DIM，不直接使用 ODS
4. ADS 层只依赖 DWS + DIM，不直接使用 DWD
5. ODS 层只能被 DWD 层引用

### 2.5 分层设计的关键原则

#### 原则一：高内聚低耦合

每一层做好自己该做的事，不越界：

- ODS 只做同步，不做清洗
- DWD 只做标准化，不做聚合
- DWS 只做汇总，不做应用定制
- ADS 只做应用适配，不做通用化

#### 原则二：避免烟囱式开发

禁止从 ODS 直接跳到 ADS 的"烟囱"链路。所有数据必须经过完整分层，确保：

- 公共逻辑沉淀在 DWD/DWS 层
- 多个 ADS 可以复用同一 DWS 表
- 数据口径统一、可追溯

#### 原则三：分层引用严格管控

| 层级 | 可引用的层 |
|------|-----------|
| ODS | 无（数据源直接同步） |
| DWD | ODS、DIM |
| DWS | DWD、DIM |
| ADS | DWS、DIM |
| DIM | ODS |

#### 原则四：适度冗余换取性能

在 DWD 层通过维度退化适当冗余维度字段，在 DWS 层构建宽表合并多个指标，都是用适度的存储冗余换取查询性能和开发效率的体现。
